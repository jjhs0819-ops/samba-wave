# Speed Optimization — 삼바웨이브 속도 최적화 총괄 전문가

## 역할

삼바웨이브의 **수집(소싱) → 전송(마켓) → 동기화(오토튠)** 전 파이프라인에서 속도와 관련된 모든 설정을 총괄 관리한다.

핵심 책임:
- 소싱처/판매처별 **속도 제한 레지스트리** 유지 (references/ 폴더)
- 속도 이슈 발생 시 **병목 분석 → 개선안 제시**
- 코드 변경 시 **레지스트리 동기화** (자기 진화)

## 병목 분석 프레임워크

속도 이슈가 보고되면 아래 순서대로 체크한다:

### 1단계: 폴링 간격 (확장앱 기반 수집인 경우)
- 확장앱 setInterval 간격, alarm 주기, 집중폴링 간격 확인
- 폴링 대기가 전체 시간의 50% 이상이면 → 간격 단축 or 즉시 트리거 도입

### 2단계: 고정 대기 (sleep/wait)
- `await wait(N)`, `asyncio.sleep(N)`, `time.sleep(N)` 검색
- 고정 대기를 **동적 감지**(DOM 폴링, 응답 확인)로 교체 가능한지 평가

### 3단계: 동시성 (Semaphore)
- `Semaphore(N)` 값이 병목인지 확인
- Cloud Run 메모리(2Gi) 대비 안전한 상한 계산
- 경험적 한계: sem ≤ 50 (일반), sem ≤ 100 (가벼운 API 호출)

### 4단계: 네트워크/프록시
- 프록시 로테이션 여부, 타임아웃 설정 확인
- 429/403 응답 빈도 → 백오프 전략 적절성 평가

### 5단계: DB/직렬화
- N+1 쿼리, 동기 DB 조회가 병목인지 확인
- 배치 처리, 캐싱으로 개선 가능한지 평가

## 개선안 제시 패턴

모든 개선안은 3단계로 제시한다:

### 안전 범위 (기존 대비 부작용 최소)
- 기존 값의 ±50% 조정
- 예: sem 15 → 20, delay 3초 → 2초
- 차단 위험 낮음, 즉시 적용 가능

### 공격적 범위 (성능 우선, 모니터링 필요)
- 기존 값의 2~3배 조정
- 예: sem 15 → 50, delay 3초 → 0.5초
- 429 모니터링 필수, 롤백 계획 수립

### 한계치 (이론적 최대, 차단 위험 존재)
- Chrome/Cloud Run/API의 물리적 한계
- 예: Chrome alarm 최소 30초, setInterval은 SW 활성 중만 동작
- 구조 변경(WebSocket, 이벤트 구동) 필요할 수 있음

## 속도 제한 유형별 가이드

### Semaphore (동시성 제어)
- **용도**: 동시 HTTP 요청 수 제한
- **조정 시 고려**: Cloud Run 메모리, 대상 사이트 차단 정책, 요청 무게(이미지 vs JSON)
- **모니터링**: 429/403 응답 비율로 적정값 판단

### Delay (요청 간 대기)
- **용도**: 연속 요청 사이 쿨다운
- **조정 시 고려**: 대상 사이트의 rate limit 정책, 로봇 감지 수준
- **최적화**: 고정 delay → 적응형(성공 시 감소, 실패 시 증가)

### 지수 백오프 (Exponential Backoff)
- **용도**: 429/차단 시 점진적 재시도
- **패턴**: base × 2^retry (예: 1초 → 2초 → 4초)
- **최대 재시도**: 보통 3~5회

### 폴링 간격 (확장앱)
- **용도**: 서버 큐에서 작업 수신
- **한계**: Chrome MV3 alarm 최소 30초, setInterval은 SW 활성 중만
- **최적화**: 이벤트 구동(tabs.onUpdated), 동적 DOM 감지

## 새 사이트 추가 시 초기 설정 가이드

### 소싱처 추가
1. 유사 사이트의 현재 설정을 reference에서 확인
2. 초기값: sem 5, delay 1초, 타임아웃 30초 (보수적 시작)
3. 테스트 후 점진 조정: 100건 → 500건 → 1000건 수집 성공 확인
4. 429/403 발생 시 delay ↑ 또는 sem ↓

### 판매처 추가
1. API 문서에서 공식 rate limit 확인 (가장 중요)
2. 공식 한도의 80%를 초기 목표로 설정
3. 이미지 업로드는 별도 sem (보통 3~5)
4. 등록 후 안정화 대기 필요 여부 테스트

## 자기 진화 규칙

**속도 관련 코드가 수정되면, 반드시 이 스킬의 references도 업데이트해야 한다.**

- sem/delay 값 변경 → 해당 사이트의 reference 레코드 업데이트
- 새 소싱처/마켓 추가 → reference에 새 레코드 추가
- 차단/429 대응 로직 변경 → 해당 사이트 비고란 업데이트
- 확장앱 폴링 간격 변경 → autotune-and-extension.md 업데이트
- 오토튠 동시성/인터벌 변경 → autotune-and-extension.md 업데이트

## 주의사항

- 속도 최적화는 **안정성과 트레이드오프**. 빠르게 만들수록 차단 위험 증가
- 소싱처는 **IP 기준 차단**이 많음 → 프록시 로테이션 고려
- 판매처는 **API 키 기준 rate limit**이 많음 → 공식 한도 준수
- Cloud Run은 **요청당 과금** → 불필요한 재시도는 비용 증가
- 확장앱은 **Chrome MV3 제약** → SW 자동 종료, alarm 최소 30초
