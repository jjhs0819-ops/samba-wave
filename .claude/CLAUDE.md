## 브랜치 기반 파일 수정 제한 [중요]
- 현재 브랜치가 `feature/XXX-plugin`이면 → `plugins/markets/XXX.py`와 `proxy/XXX.py`만 수정
- 현재 브랜치가 `feature/XXX-sourcing`이면 → `plugins/sourcing/XXX.py`와 `proxy/XXX.py`(또는 `proxy/XXX_sourcing.py`)만 수정
- 위 파일 외 다른 파일 수정이 필요하면 → 수정하지 말고 "이 파일도 수정이 필요합니다. 팀장에게 확인하세요"라고 안내
- `__init__.py`, `market_base.py`, `sourcing_base.py`, `collector/` 폴더는 절대 수정 금지

## 팀원 PR 머지 규칙 [최우선]
- 팀원은 자기 담당 플러그인 파일만 수정해야 함 (plugins/, proxy/ 내 본인 파일)
- 그 외 모든 파일은 공유 파일이므로, 팀원 PR에 포함되면 안 됨
- 머지 전 반드시 `git diff main...팀원브랜치 --name-only`로 변경 파일 목록 확인
- 담당 플러그인 외 파일이 변경됐으면 → 컨플릭트 없어도 바로 머지 금지
- 해당 파일의 diff를 보고하고 팀장 승인을 받을 것
- 컨플릭트가 없다고 안전한 게 아님 — Git은 한쪽만 변경하면 자동 채택하므로 팀장 코드가 통째로 유실될 수 있음 [중요]

## 푸시 전 필수 [최우선]
- 백엔드 Python 파일 수정 시 **반드시** `cd backend && .venv/Scripts/python.exe -m ruff format .` 실행 후 푸시
- `ruff format` 누락 = CI 빌드 실패. git add 전에 무조건 실행할 것
- `ruff check --fix .`도 함께 실행
- 로컬 빌드(`pnpm build`)는 불필요 — CI + Vercel이 검증함

## 코드 반영 규칙 [중요]
- 요청 분석 후 변경 내용을 먼저 설명만 할 것
- "반영해줘" / "적용해줘" / "ㅇㅇ" / "ㅇㅋ" 등 명시적 승인이 있을 때만 실제 파일 수정
- 분석/조사/원인파악 요청은 코드 수정 없이 결과만 보고
- 푸시("푸시" / "push")도 코드 수정과 동일하게 승인 후 실행 — 승인 없이 코드 수정 후 바로 푸시 금지 [최우선]
