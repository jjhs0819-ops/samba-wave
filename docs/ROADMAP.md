# 삼바웨이브 개발 로드맵

> 최종 목표: 상품 10만건, 소싱처 20개, 판매처 30개, 개발자 5명 병렬 개발

---

## 현재 상태

| 항목 | 현재 | 목표 |
|------|------|------|
| 상품 수 | 수천 건 | 10만 건 |
| 소싱처 | 1개 실사용 (무신사) | 20개 |
| 판매처 | 4-5개 | 30개 |
| 개발자 | 1명 | 5명 |
| 코드 구조 | 단일 파일 3000줄 | 플러그인 × N개 (각 200-300줄) |

### 주요 병목

- `collector.py` 3035줄 — 모든 수집 로직이 한 파일
- `products/page.tsx` 3147줄 — 모든 상품 UI가 한 파일
- 소싱/마켓 추가 시 기존 코드 수정 필요 → 충돌 불가피
- 10만건 시 프론트 렌더링, DB 쿼리 모두 병목

---

## Phase 1: 기반 정리 (4주)

> 5명이 동시에 작업 가능한 구조 만들기

- [ ] `backend/domain/samba/plugins/sourcing_base.py` — 소싱 플러그인 인터페이스
- [ ] `backend/domain/samba/plugins/market_base.py` — 마켓 플러그인 인터페이스
- [ ] 무신사를 첫 번째 소싱 플러그인으로 추출
- [ ] 스마트스토어를 첫 번째 마켓 플러그인으로 추출

### 1-2. 프론트엔드 파일 분리 (Week 2)

## 핵심 원칙

### 1. 플러그인 = 독립 파일
```
새 소싱처 추가 = 파일 1개 추가 (기존 코드 수정 0)
새 마켓 추가 = 파일 1개 추가 (기존 코드 수정 0)
→ 5명이 충돌 없이 개발 가능
```

### 2. 작업 큐
```
현재: API 요청 → 수집 완료까지 대기 (타임아웃 위험)
목표: API 요청 → 큐에 넣기 → 즉시 응답 → 워커가 처리
```

### 3. 10만건 = DB가 핵심
```
프론트: 가상 스크롤 (한번에 50건만 렌더)
백엔드: 커서 페이지네이션 (OFFSET 금지)
캐시: Redis 30초 TTL
검색: PostgreSQL full-text search
```

### 4. 코드 규칙
```
파일당 최대 500줄
함수당 최대 50줄
PR 머지 전 최소 1명 리뷰
테스트 없는 플러그인 머지 금지
```
